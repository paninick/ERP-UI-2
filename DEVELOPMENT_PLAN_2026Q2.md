## 二、开发优先级判定原则

按以下顺序判定优先级（来自 LONG_TERM_PRODUCTION_MODEL_PLAN.md）：

1. 是否是上游真实（source truth）
2. 下游模块是否已依赖它
3. 是否减少未来重写风险
4. 是否快速建立生产信任
5. 是否支撑后续成本正确性

### Wave 1：主数据基础（第2-4周）

**目标**：让所有业务模块的选择器来自后端，不再依赖硬编码

执行顺序（不可调换）：

### Wave 2：业务数据补齐（第5-7周）

## 四、DeepSeek v4 可用任务规范

### 当前档位

```
档位：可调教（接近受控可用）
主线编码资格：否
候选补丁资格：是
```

### 可分配的任务类型

DeepSeek v4 只能承接以下类型任务：

| 类型 | 说明 |
|---|---|
| 单文件 SQL 修复 | 白名单明确，幂等性可验证 |
| 单文件后端补丁 | 白名单明确，mvn compile 可验证 |
| 单文件前端补丁 | 白名单明确，npm build 可验证 |
| 只读分析 | 读取指定文件，输出风险清单 |

### 禁止分配的任务类型

| 类型 | 原因 |
|---|---|
| 多文件跨模块修改 | 历史记录显示容易越界 |
| 审批流设计 | 需要架构判断，不适合候选模型 |
| 状态文档更新 | 一票否决项 |
| 新模块全栈交付 | 超出当前档位 |
| 数据库直接执行 | 需要主线权限 |

### 每次任务必须包含的要素

```
1. 明确的白名单文件列表（不超过3个文件）
2. 明确的禁止文件列表
3. 明确的验证命令（MySQL/mvn/npm 其中之一）
4. 明确的完成判定标准
5. 明确的一票否决项
```

### 任务模板（DeepSeek v4 专用）

```
你当前处于模型测试阶段，不具备主线编码权限。

本轮身份：候选修复者
本轮目标：[具体目标，一句话]

开始前，你必须先输出测试许可单（8项）。

允许修改的文件：
1. [文件路径]

禁止修改的文件：
1. 所有项目状态文档（PROJECT_PROGRESS.md、PROJECT_ISSUES.md 等）
2. 所有治理文档
3. 白名单外的所有文件

允许执行的命令：
1. [验证命令]

禁止执行的命令：
1. 启动服务
2. 杀进程
3. 修改环境
4. 推进下一任务

强制要求：
1. 必须真实提交补丁
2. 补丁必须直接命中 [具体问题]
3. 必须执行验证命令

输出要求：
1. 第一段只输出测试许可单
2. 修改后只输出改动清单、命中点说明、验证结果、未验证项、风险
3. 不得输出"已完成""可进入下一任务"

越界即失败。
```

### DeepSeek v4 升档条件

从"可调教"升为"受控可用"需同时满足：

1. 连续 5 轮零越界（当前已有 Round 2E 基本通过，需继续积累）
2. 至少 2 个真实补丁经 Claude 复审认为方向正确
3. 命中率 ≥ 80%
4. 一致性 ≥ 75%
5. 验证意识 ≥ 75%

## 七、验证标准总则

每个模块交付必须满足（不可降级）：

```
1. dict        字典已录入，无硬编码枚举
2. sql         表结构已建，幂等可重跑，索引/唯一键已定
3. domain      Java domain 含所有业务字段，toString() 完整
4. mapper      XML 含 resultMap/SELECT/INSERT/UPDATE，字段与 domain 一致
5. service     含校验/状态流转/审批日志写入
6. controller  含业务动词接口，@PreAuthorize 已注册
7. frontend api 与后端路由 1:1 对应，无假接口
8. frontend page 含列表/表单/状态标签/源头只读字段
9. approval/print/scan 已明确是否需要，已实现
10. live verify 真实字典值 + 真实后端响应 + 真实保存记录 + 异常路径
```
